## 1. 这不是“模型能力问题”，本质是“执行系统设计问题”

你提到的理解实验、写代码、跑命令、写报告，这些单点能力现在都不算最难。真正难的是这四件事之间的**闭环控制**：

1. 正确读取实验说明和验收标准
2. 在远程环境中执行操作
3. 判断当前结果是否满足验收条件
4. 失败后继续迭代，而不是提前停机或跑偏
5. 最后把全过程沉淀成**可审计**的正式报告

所以项目核心不是“给模型一个 shell”，而是设计一个：

**可观测、可约束、可恢复、可验收、可留痕** 的实验执行代理系统。

这点你已经意识到了，但还可以更明确一些：
**ssh/sshfs/tmux 都只是 I/O 层，不是系统核心。**

## 2. 你当前路线里，最合理的部分

你现在的几个前提，基本都对。

### 2.1 单实验聚焦

这个非常重要。
SEED 实验虽然规范，但不同实验的知识类型差异很大：Web 安全、系统安全、网络、密码学、容器、漏洞利用，策略空间差异很大。
所以第一版系统不该追求“通用”，而应追求：

**在一个实验上打通完整闭环。**

建议目标定义成：

* 输入：实验文档 + labsetup + 目标 VM 连接信息
* 输出：可复现实验结果 + 过程日志 + 验收结论 + 正式报告

### 2.2 提前提供文档和 labsetup

这也是对的。
因为这相当于把开放世界问题收缩成半封闭任务。Agent 不需要自己到处猜环境，而是从明确输入开始工作。

你实际上是在给 Agent 提供三类上下文：

* **任务上下文**：实验指导、目标、验收标准
* **环境上下文**：docker-compose、模板代码、目录结构
* **执行上下文**：SSH 可达、目标 VM 已准备好

这会显著提高成功率。

### 2.3 人工预先打通 SSH

也对。
“让 Agent 自己去解决网络、认证、密钥分发、跳板机、KnownHosts、端口映射”等等，会把问题复杂度抬高一个量级。
第一阶段把这些基础设施前置配置好，是合理的工程简化，不算过度人工干预。

## 3. 你目前设计里最不完美的地方

核心问题有四个。

---

### 3.1 不建议把 **sshfs** 作为主路径

你想到 sshfs，是因为想让本地 Agent “像操作本地目录一样操作远程文件”，这在直觉上很自然，但它更适合作为**辅助手段**，不适合作为主执行通道。

原因：

#### 问题 1：文件视图不等于执行视图

实验完成与否，不取决于文件改得对不对，而取决于：

* 命令是否按顺序执行
* 服务是否真正启动
* 容器/进程/网络状态是否正确
* 输出结果是否满足验收条件

sshfs 只能解决“文件同步”，不能天然解决“远程进程状态管理”。

#### 问题 2：一致性和异常处理很脆

sshfs 在以下场景容易变得难控：

* 网络抖动
* 大量小文件变更
* 远程文件锁
* 容器 bind mount 与宿主路径交互
* 权限 / uid gid 映射
* 某些程序原子写临时文件再 rename

这些问题一旦出现，Agent 很难判断是“实验失败”还是“挂载层异常”。

#### 问题 3：让控制面和数据面耦合

你本质需要两种通道：

* **控制面**：发命令、看退出码、拿 stdout/stderr、查进程状态
* **数据面**：读写代码、配置、结果文件、截图、pcap、日志

sshfs 只解决数据面，不解决控制面。
如果把主系统建立在 sshfs 上，你后面还是得额外补一个稳健的控制面。

#### 结论

**sshfs 可以有，但不该是核心依赖。**
更合适的位置是：

* 方便本地查看结果文件
* 方便归档日志和实验产物
* 方便报告撰写阶段读取附件

而不是作为 Agent 的主工作流基础。

---

### 3.2 “ssh + tmux + 重定向输出到文件”能用，但不够结构化

这比 sshfs 更靠谱，但还不够。

你这个思路的价值在于：

* 远程命令可以长期运行
* 会话不断线
* 输出可以持续采集
* 可以断点恢复

这些都很好。

但问题是 tmux 输出本质还是**面向人类终端的文本流**，不是面向 Agent 的结构化执行记录。
会遇到这些问题：

* 如何精确区分每条命令的开始/结束？
* 如何获取退出码？
* 如何判断某段输出属于哪个子任务？
* 如何避免 shell prompt、颜色控制字符、分页器污染？
* 如何处理交互式程序？
* 如何知道命令卡住了还是还在跑？

tmux 更适合“维持一个人类可回看的长生命周期会话”，但不是最好的 Agent 执行协议。

#### 更好的做法

你应该把远程执行抽象成一个**受控命令执行器**，而不是直接把 tmux 当协议层。

例如定义一个远程 runner，支持：

* `run(command, cwd, timeout, env)`
* 返回：

  * stdout
  * stderr
  * exit_code
  * start_time / end_time
  * artifacts
* 支持：

  * 流式日志
  * 超时终止
  * 后台任务
  * 状态查询
  * 会话恢复

换句话说：

**tmux 可以作为底层实现细节，但上层不应该直接面向 tmux 编排。**

---

### 3.3 你现在还缺一个“验收器”

这是你整个设想里最关键、但还没被充分结构化的部分。

SEED 实验能否真正自动化，不取决于 Agent 会不会操作 shell，而取决于：

> 它能不能在没有人类追问的情况下，自己判断“任务完成了没”。

你已经提到“不能做一半问用户接下来怎么办”，这本质上就是在说：
系统必须有**终止判定机制**。

而这个终止机制不能只靠 prompt。

#### 为什么单靠 prompt 不够

因为 prompt 只能影响风格，不能保证事实判断。
模型可能会：

* 误以为成功
* 漏掉一个验收项
* 忘记截图或保存证据
* 看到部分结果就提前结束

#### 正确做法

把实验文档中的“验收标准”抽取成机器可检查的对象。
即做一个 **Acceptance Checker / Verifier**。

它至少要能检查：

* 目标文件是否存在
* 目标服务是否启动
* 指定命令输出是否包含某些关键证据
* 某些攻击/防御效果是否成立
* 是否产生规定截图/日志/实验结果文件
* 报告中是否覆盖了规定章节

如果能做到，Agent 的循环就从：

> 想一想 → 操作一下 → 好像差不多了 → 结束

变成：

> 计划 → 执行 → 收集证据 → 调用验收器 → 若不通过则继续

这是质变。

---

### 3.4 报告撰写不能只是“最后总结一下”

报告部分建议你单独设计，不要当作最后一步顺手生成。

因为正式报告通常要求：

* 实验目的
* 环境说明
* 关键步骤
* 结果展示
* 对现象的解释
* 验收结论
* 可能还有反思 / 安全分析 / 原理说明

如果没有全过程证据链，最后模型很容易“脑补”，报告会变成不可靠的后验文学。

所以报告系统应该吃的是一个**结构化实验记录**，而不是仅仅吃 shell 输出。

建议全程维护一个 experiment trace，例如：

* task_id
* step_id
* intent / rationale
* command
* stdout/stderr 摘要
* exit_code
* generated_files
* screenshots
* checkpoints
* acceptance evidence
* final conclusions

最后报告由这个 trace 生成。这样报告才可追溯。

## 4. 我建议的更优技术路线

### 总体原则

不要把项目定义成“一个能 ssh 的 codex 魔改版”。
应该定义成：

> 一个面向 SEED 实验的 agentic execution system

系统分五层。

---

### 第一层：任务标准化层

把每个实验统一表示成一个 task spec。

例如：

* 实验名称
* 实验目标
* 前置条件
* 环境启动方式
* 验收标准
* 限制条件
* 预期产物
* 报告模板

这个 spec 最好是结构化的，哪怕最开始是人工从文档里抽出来也值得。

例如一个 JSON / YAML：

```yaml
name: buffer_overflow_lab
goal:
  - gain root shell inside target container
inputs:
  - seed_lab_manual.pdf
  - labsetup.zip
environment:
  startup:
    - docker compose up -d
acceptance:
  - command: "id"
    must_contain: "uid=0(root)"
  - file_exists: "/tmp/exploit.py"
artifacts:
  - screenshot: true
  - logs: true
report_sections:
  - objective
  - environment
  - methodology
  - results
  - analysis
  - conclusion
```

这一步很关键。
否则你的 Agent 永远在直接啃自然语言文档，稳定性会差很多。

---

### 第二层：远程执行层

这一层不建议直接暴露 shell，而建议封装成工具接口。

最小工具集建议是：

* `read_file(remote_path)`
* `write_file(remote_path, content)`
* `append_file(remote_path, content)`
* `list_dir(remote_path)`
* `run_command(command, cwd, timeout, env)`
* `start_background_job(command, cwd, env)`
* `get_job_status(job_id)`
* `kill_job(job_id)`
* `fetch_artifact(remote_path, local_path)`
* `capture_terminal_snapshot(job_id or session_id)`

底层实现可以是：

* SSH exec
* SFTP / SCP
* 必要时 tmux / nohup / systemd-run 维护后台任务

但上层 Agent 不直接和 tmux 语义耦合。

#### 为什么这样更好

因为模型看到的是稳定 API，而不是脆弱的 shell 细节。
这对“完成一个任务再结束”非常重要。

---

### 第三层：实验工作流层

这一层是 agent loop。

建议固定成如下状态机，而不是自由聊天式代理：

1. **Parse**：解析实验说明和 task spec
2. **Plan**：生成子任务列表
3. **Setup**：检查环境、启动 lab
4. **Execute**：逐步操作
5. **Verify**：运行验收器
6. **Repair**：失败则定位问题并修复
7. **Finalize**：收集产物
8. **Report**：生成正式报告

也就是一个典型的：

**Plan → Act → Observe → Verify → Iterate**

而不是“让模型随便发挥”。

---

### 第四层：监督与终止层

这个就是你提到的 main agent / 外部监督器。

我建议：**必须有**，而且不要只靠单智能体。

最简单的方式不是搞多智能体大编排，而是做一个**双层结构**：

#### 角色 A：Worker

* 真正执行命令
* 改代码
* 启服务
* 收集结果

#### 角色 B：Supervisor / Verifier

* 审查 Worker 当前状态
* 判断是否满足验收标准
* 判断是否缺少证据
* 决定继续执行还是进入报告阶段

这样比“一个 Agent 靠 prompt 自觉”稳得多。

你不一定非要两个独立模型实例，也可以是：

* 同一模型，不同 system prompt，不同调用阶段
* 或一个固定程序化 verifier + 一个 LLM supervisor

工程上更推荐后者：
**程序化 verifier 负责硬约束，LLM supervisor 负责软判断。**

---

### 第五层：证据与报告层

建议从第一天就做日志与证据结构化，不要后补。

建议至少保存：

* 每条命令
* 退出码
* 原始输出
* 关键中间文件
* 截图
* 环境信息
* 失败重试记录
* 最终验收结果

报告可以分两版：

#### A. 正式版

给导师/课程验收看，强调：

* 目标
* 环境
* 操作步骤
* 结果截图
* 分析
* 结论

#### B. 审计版

给开发者看，强调：

* 全命令日志
* 工具调用记录
* 文件 diff
* 验收器输出
* 失败路径

这两版最好分离。

## 5. Skill 适不适合做？

结论：**适合做，但不要一开始就把“完整实验代理系统”硬塞成一个 Skill。**

### 更合理的定位

Skill 更适合封装：

* 固定工作流
* 固定输入约定
* 固定输出格式
* 固定注意事项
* 固定禁止事项

也就是把“实验执行规范”固化下来。

但 Skill 不适合承载太多**底层控制逻辑**。
真正的远程执行、任务状态管理、日志留存、验收器，最好是外部系统或本地工具层完成。

#### 建议分层

* **Skill**：做任务指导层 / 策略层 / workflow policy
* **Local tool / wrapper**：做 SSH、SFTP、runner、artifact 管理
* **Main agent**：做循环控制与监督

这样未来即使底层模型换了、CLI 换了、Codex 换了，Skill 仍然可复用。

这是你说的“复用性强，不容易随模型迭代淘汰”的真正实现方式。

---

## 6. “确保完成再结束”应该怎么做

这个问题不要寄希望于提示词。

提示词只能提高概率，不能提供保证。

建议使用三重约束：

### 6.1 明确的完成条件

必须程序化表示。
没有通过 acceptance checker，不允许结束。

### 6.2 有限状态机

Agent 只能处在：

* planning
* executing
* verifying
* repairing
* reporting
* done

不允许在中途进入“等待用户进一步指示”。

### 6.3 失败预算与回退机制

比如：

* 每个子目标最多重试 N 次
* 连续 N 次失败则进入诊断模式
* 诊断模式输出当前障碍、证据、下一步可执行方案
* 而不是无限乱试或向不存在的用户求助

这样系统至少是“受控失败”，不是“混乱失败”。

## 7. 一个更具体的最小可行方案

我建议你的 MVP 不要直接追求完整平台，先做下面这个版本：

### MVP 目标

只支持一个固定 SEED 实验，比如某个最典型实验。

### 输入

* 实验文档
* labsetup.zip
* VM SSH 信息
* 预定义的 task spec

### 组件

1. **实验解析器**

   * 从实验文档抽取目标、步骤、验收标准
   * 前期可半人工

2. **远程执行器**

   * SSH 执行命令
   * SFTP 传文件
   * 后台任务管理
   * 日志归档

3. **Agent loop**

   * 计划
   * 执行
   * 观察
   * 修复
   * 验收

4. **验收器**

   * 若干规则检查器
   * 命令输出匹配
   * 文件存在性检查
   * 关键结果截图检查

5. **报告生成器**

   * 基于 trace 自动生成 markdown / latex / docx

### 暂时不要做的

* 通用多实验自动解析
* 复杂多智能体社会
* 完全无人工的环境发现
* 过度依赖 sshfs
* 过度追求 Skill 化

先把闭环跑通。

## 8. 对你当前几个具体问题的直接回答
